# HabitFlow — контекст для разработки и ИИ-агентов

Этот файл держит **устойчивый контекст** между сессиями чата: новый агент не видит прошлые диалоги. Обновляйте документ после заметных изменений в продукте, архитектуре или процессе деплоя.

---

## Продукт и стек

- **Что это:** офлайн PWA — трекер привычек, данные в `localStorage`, без бэкенда.
- **UI:** статический фронт — `index.html`, `css/style.css`, логика в `js/*.js` (точка входа по смыслу — `js/ui.js` плюс `render.js`, `actions.js`, `state.js`, `data.js`, …).
- **PWA:** `manifest.json`, `sw.js`, регистрация SW в `js/ui.js` (после `DOMContentLoaded` / `load`).
- **Язык интерфейса:** основной репозиторий — **RU**; второй репозиторий исторически синхронизируется с EN-вариантом строк (см. ниже).

---

## Два репозитория GitHub

| Remote        | Репозиторий        | Назначение   |
|---------------|--------------------|--------------|
| `origin`      | `habitflow-ru`     | основная ветка `master` |
| `habitflow-en`| `habitflow-en`     | зеркало/EN, ветка `master` |

**Важно:** истории `master` на двух репозиториях **могут расходиться** (разные хеши при тех же правках). Прямой `git merge` одного в другой давал **массовые конфликты** (RU vs EN в строках UI).

**Рабочая схема синхронизации фич в EN-репо:**

```text
git fetch habitflow-en
git checkout -b sync-en habitflow-en/master
git cherry-pick <hash-коммита-с-ru-master>
# при конфликтах — править вручную под EN-строки, если они отличаются
git push habitflow-en sync-en:master
git checkout master
git branch -d sync-en
```

После пуша в `origin` дублируйте нужные коммиты в `habitflow-en` таким образом.

## Изменения, зафиксированные в ходе сессии (и связанные коммиты)

Ниже — то, что было сделано **в одной линии разработки** (описание + коммиты на `habitflow-ru`; на EN те же патчи попадали отдельными хешами после cherry-pick).

### 1. Установка PWA — кнопка и диалог

- **Задача:** дать пользователю кнопку, по которой открывается **системный диалог установки** (где браузер поддерживает это).
- **Реализация:**
  - Перехват `beforeinstallprompt` с `preventDefault()`, сохранение события, по клику — `prompt()` и обработка `userChoice`.
  - Событие `appinstalled`: сброс deferred prompt, флаг «только что установили» для текста, пока вкладка ещё не в `standalone`.
  - **Онбординг:** шаг с установкой (в массиве шагов `_obSteps` в `js/ui.js`, индекс шага установки — `_OB_INSTALL_STEP === 6`): блок `.ob-install-block`, кнопка `#obPwaInstallBtn`, fallback-текст `#obPwaInstallFallback`.
  - **Меню ☰:** пункт «Установить приложение» → `closeBurger(); pwaTryInstall()`.
  - Стили: `.ob-install-block`, `.ob-btn-install`, `.ob-install-fallback` в `css/style.css`.
- **Ограничения:**
  - **`beforeinstallprompt` нет в Safari на iOS** — там только путь «Поделиться» → «На экран „Домой“».
  - Нужны **HTTPS** (или localhost) и корректный manifest + SW.

**Коммит (RU):** `30058b8` — `feat(pwa): кнопка установки приложения в онбординге и меню`

### 2. iPhone: баннер «откройте в Safari» (ВК и др.)

- **Задача:** если ссылку открыли **во встроенном браузере** (например ВК), установка/«Домой» часто недоступны — показать подсказку **только на iOS**, для Android не показывать.
- **Реализация:**
  - Распознавание известных приложений по **User-Agent** (`VKJSBridge`, `VK/…`, Instagram, FB, Telegram, TikTok, … — см. `_iosInAppHostLabel` в `js/ui.js`).
  - Дополнительная эвристика **`_iosLikelyEmbeddedWebView`:** iOS, не `CriOS`/`FxiOS`/`EdgiOS`, нет типичной цепочки `Version/… Mobile/… Safari/…` как у полноценного Safari.
  - Не показывать в режиме **установленного PWA** (`standalone` / `navigator.standalone`).
  - Закрытие крестиком → `sessionStorage` ключ `habitflow_ios_inapp_hint` = `1` на сессию.
  - Разметка: `#iosInAppBanner` в `index.html`; стили `.ios-inapp-banner*`, отступы для `.app` и `.ob-overlay` при `html.ios-inapp-banner-visible` в `css/style.css`.
- **Ограничение:** UA может меняться; ложные срабатывания/пропуски возможны — список маркеров расширяется в коде при необходимости.

**Коммит (RU):** `d6e9669` — `feat(ios): баннер для встроенного браузера (ВК и др.) с подсказкой открыть в Safari`

## Рекомендации для агента и человека

1. Перед крупной правкой — прочитать этот файл и при необходимости `git log -10`.
2. После значимой фичи — **одна строка или абзац** в разделе «Изменения» + обновить «Где искать» при новых файлах.
3. Два репозитория: не полагаться на автоматический merge; по умолчанию **cherry-pick** с `habitflow-en/master`.
4. Не раздувать правила в `.cursor/rules` дублированием всего AGENTS — в правилах лучше краткие стабильные ограничения (язык, «не трогать без запроса»), детали — здесь.
